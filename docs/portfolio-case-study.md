# AntiADHD 포트폴리오 케이스 스터디

## 문제

일정 앱은 계획을 기록하게 해주지만, 집중이 어려운 사용자가 `무엇부터 시작할지` 결정하는 비용은 줄여주지 못한다. AntiADHD는 일정 등록보다 계획을 작은 행동으로 분해하고 실제 시간에 배치해 실행하는 흐름에 초점을 맞췄다.

## 해결

- AI 작업 분해: 큰 할 일을 예상 시간과 함께 시작 가능한 단계로 변환
- 음성 일정 입력: 자연어 음성을 구조화하되 저장 전 제목과 시간을 사용자가 확인
- 실행 루프: 일정, 포커스 세션, 루틴, 목표, 하루 회고를 연결
- 운영 기반: Docker 이미지, k3s, PostgreSQL, GitHub Actions, Prometheus, Grafana 구성

## 대표 운영 장애와 복구

운영 DB에는 과거 Hibernate가 만든 외래 키 이름이 남아 있었고, 새 Flyway 마이그레이션은 개발 환경의 이름을 가정했다. 새 Backend Pod는 마이그레이션에 실패해 CrashLoop에 진입했다.

1. Pod 로그와 Actuator 상태로 애플리케이션 코드가 아닌 스키마 마이그레이션 실패임을 확인했다.
2. 직전 GHCR 이미지로 롤백해 서비스부터 복구했다.
3. 운영 스키마 복제본에서 트랜잭션 기반 dry-run을 수행했다.
4. 외래 키 이름 대신 테이블·컬럼 관계로 제약 조건을 탐색하도록 마이그레이션을 수정했다.
5. 재배포 후 readiness, API smoke test, Android 전체 사용자 여정을 검증했다.

이 경험을 통해 마이그레이션은 신규 스키마뿐 아니라 실제 레거시 스키마에서 검증해야 하며, 배포 자동화와 롤백 가능성이 한 세트라는 점을 확인했다.

## 검증 근거

- Backend 자동 테스트 28개
- Mobile 단위 테스트 및 Maestro Android E2E
- Flyway 마이그레이션 7개
- GitHub Actions 이미지 빌드·보안 검사
- Prometheus 수집과 Grafana 운영 대시보드

## 정직한 현재 상태

AntiADHD는 본인과 파트너가 사용하는 2인 비공개 베타다. 단일 노드 홈 서버는 대규모 트래픽이나 고가용성을 주장하기 위한 구성이 아니라, 제한된 자원에서 배포·관측·백업·장애 복구의 전체 수명주기를 직접 경험하기 위한 선택이다.
